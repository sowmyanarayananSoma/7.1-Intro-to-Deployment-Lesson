# 7.1 – Section 3: Preparing for Deployment

This section is a practical guide. Before you deploy any app, these are the things you need to have in place — every time.

---

## 1. Version Control

Your code must be in a Git repository before anything else can happen. Deployment platforms pull your code directly from GitHub. Without version control, there is no deployment pipeline, no rollback, and no safe way for multiple people to work on the same codebase.

If you haven't already:

```bash
git init
git add .
git commit -m "initial commit"
git remote add origin https://github.com/your-username/your-repo.git
git push -u origin main
```

This is the foundation everything else builds on.

---

## 2. Branching Strategy

One rule covers everything: `main` is always production. You never commit to it directly. All work flows through two stepping stones first — `develop` then `staging` — before anything reaches real users.

---

### The three permanent branches

| Branch | Environment | Purpose |
|---|---|---|
| `develop` | Dev | All finished work lands here first |
| `staging` | Staging | Final testing before going live |
| `main` | Production | Live app — real users |

Code only moves left to right. Never backwards.

---

### How it works day to day

**1. Branch off `develop`**
```bash
git checkout develop
git checkout -b feature/add-search-bar
```

**2. Open a pull request into `develop`**

A teammate reviews it. Once merged, the dev environment updates automatically.

**3. When the team is ready, merge `develop` into `staging`**

This is a team decision — not automatic. QA testing happens here. If bugs are found, fix them in `develop` first, then update `staging` again. Never fix directly on `staging`.

**4. When staging passes, merge `staging` into `main`**

Production deploys. Done.

---

### Branch naming

| Prefix | When to use | Example |
|---|---|---|
| `feature/` | Building something new | `feature/add-search-bar` |
| `fix/` | Bug found in dev or staging | `fix/login-redirect-error` |
| `hotfix/` | Something broken in production right now | `hotfix/payment-crash` |
| `docs/` | Documentation only | `docs/update-readme` |
| `chore/` | Dependencies, config, cleanup | `chore/update-packages` |

Rules: lowercase, hyphens between words, descriptive but short, no personal names. Include a ticket number if your team uses one (`feature/GH-42-search-bar`).

---

### Hotfixes — the exception

Something is broken in production and can't wait. Branch directly off `main`, fix it, then merge back into **both** `main` and `develop`.

```bash
git checkout main
git checkout -b hotfix/payment-crash
# fix it, then:
git checkout main && git merge hotfix/payment-crash    # fixes production
git checkout develop && git merge hotfix/payment-crash # keeps develop in sync
```

If you skip merging back into `develop`, the bug comes back next time `develop` goes to staging.

---

### Release branches — optional

For most web apps, merging `staging` into `main` is enough. If you need a window to bump version numbers or write release notes without blocking `staging`, cut a short-lived branch off `staging`:

```bash
git checkout staging
git checkout -b release/v1.2.0
# final checks, then merge into main and tag it
```

Delete it after shipping. It is not permanent.

## 3. Environment Variables

Any value that changes between environments (local, staging, production) or that should never be exposed in your code belongs in an environment variable.

**What goes in `.env`:**

```bash
# Database
MONGO_URI=mongodb+srv://username:password@cluster.mongodb.net/mydb

# Authentication
JWT_SECRET=your-secret-key-here
SESSION_SECRET=another-secret

# External APIs
STRIPE_SECRET_KEY=sk_live_...
SENDGRID_API_KEY=SG....

# App config
PORT=5000
NODE_ENV=production
CLIENT_URL=https://myapp.com
```

**The rules:**

1. Never hardcode any of these values in your source code
2. Never commit `.env` to GitHub — add it to `.gitignore` immediately when you create the file
3. On your deployment platform (Render, Railway, etc.), enter these values manually through their environment variables dashboard — not in your code
4. Use `process.env.VARIABLE_NAME` to access them in your backend code
5. In Vite (React frontend), prefix variables with `VITE_` and access them with `import.meta.env.VITE_VARIABLE_NAME`

**Create your `.gitignore` before your first commit:**

```
# .gitignore
node_modules/
.env
.env.local
.env.production
dist/
build/
```

If you accidentally commit a `.env` file to a public repository, rotate every credential in it immediately. Simply deleting the file in a new commit is not enough — the credentials are still visible in the git history.

## 4. Security Considerations

### Never commit secrets
Covered above, but worth repeating. API keys, database passwords, and JWT secrets that end up in a public GitHub repo are compromised the moment they're pushed. Automated bots scan GitHub for exposed credentials continuously.

### Use HTTPS
All production apps should serve traffic over HTTPS. Most deployment platforms (Render, Vercel, Netlify) provision SSL certificates automatically — you don't have to do this manually.

### Configure CORS correctly
Your Express backend needs to explicitly allow requests from your frontend's domain. In development you may have `*` as the allowed origin — before deploying, restrict this to your actual frontend URL.

```js
// Development (too permissive for production):
app.use(cors());

// Production:
app.use(cors({
  origin: process.env.CLIENT_URL  // e.g. "https://myapp.onrender.com"
}));
```

### Set `NODE_ENV=production`
Many libraries behave differently in production mode — they disable verbose error messages, enable caching, and optimize performance. Set this as an environment variable on your deployment platform.

## 5. Deployment Scheduling and Timing

Even for small apps, it's worth thinking about when you deploy.

**Two questions to ask:**
1. When do the fewest users have the app open?
2. Are you (or a teammate) available to monitor it after pushing?

Deploying during off-peak hours reduces the number of users affected if something goes wrong. But deploying at 3am when no one is awake to fix a problem isn't smart either. Find the overlap — low traffic, someone available.

For your first deployment especially, do it when you have time to watch what happens and respond if needed.

## 6. Pre-Deployment Checklist

Before you push to production for the first time, go through this list:

```
Version Control
☐ Code is committed and pushed to GitHub
☐ You are deploying from the correct branch (main or release branch)
☐ No unfinished work-in-progress is merged into that branch

Environment Variables
☐ .env is in .gitignore
☐ .env has never been committed to GitHub
☐ All required environment variables are configured on the deployment platform
☐ No hardcoded secrets exist anywhere in the codebase

Security
☐ CORS is configured to allow only your frontend's domain
☐ NODE_ENV is set to "production" on the platform
☐ All API keys and secrets are rotated if they were ever accidentally exposed

Database
☐ Database is hosted (e.g. MongoDB Atlas), not running on localhost
☐ Connection string uses process.env.MONGO_URI
☐ Database access is restricted to your app's IP or connection string only

Build & Configuration
☐ PORT uses process.env.PORT (not a hardcoded number)
☐ Frontend API URL points to the deployed backend, not localhost
☐ npm run build completes without errors locally

Dependencies
☐ All dependencies are listed in package.json (not just installed globally)
☐ No development-only packages are required for the app to run in production
```

## 7. Key Terms

| Term | Meaning |
|---|---|
| **Pull request (PR)** | A request to merge one branch into another, including a code review step |
| **Feature flag** | A toggle in code that enables or disables a feature without redeploying |
| **Hotfix** | An urgent fix applied directly to production and merged back to other branches |
| **`.gitignore`** | A file that tells Git which files to never track or commit |
| **CORS** | Cross-Origin Resource Sharing — controls which domains can make requests to your backend |
| **SSL/HTTPS** | Encrypted connection between the browser and your server |
| **`NODE_ENV`** | An environment variable that tells Node.js whether it's running in development or production |
