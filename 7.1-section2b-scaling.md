# 7.1 – Section 2 (continued): Scaling in the Cloud

---

## 7. Why Scaling Matters

When your app is running on localhost, it serves one user: you. Once it's deployed and real people are using it, you have to think about what happens when traffic grows.

**Scaling** is the process of adjusting your infrastructure to handle more (or less) load — more users, more requests, more data.

There are two fundamental ways to scale a system, and understanding the difference is important for how you design and deploy apps.

---

## 8. Vertical Scaling (Scaling Up)

<invoke name="visualize:show_widget">Vertical scaling means making your existing server more powerful. Instead of adding more machines, you upgrade the one you have — more CPU cores, more RAM, faster storage.

```
Before scaling:           After scaling up:
┌─────────────┐           ┌─────────────┐
│  Server     │           │  Server     │
│  2 CPU      │  ──────►  │  8 CPU      │
│  4 GB RAM   │           │  32 GB RAM  │
│  50 GB SSD  │           │  500 GB SSD │
└─────────────┘           └─────────────┘
```

### How it works in practice

On a cloud platform like AWS, Azure, or Render, you go into your dashboard and select a larger instance type. The platform stops your server, moves it to a more powerful machine, and restarts it. Your app code doesn't change at all.

### When it works well

- Your app isn't designed to run as multiple copies (e.g. a relational database with complex transactions)
- You need a quick fix for a performance problem
- Your traffic is moderate and predictable

### The hard ceiling

Every machine has a maximum. You can only add so much CPU and RAM to a single server before you hit the physical limit of what exists. At that point, vertical scaling stops being an option.

Vertical scaling also creates a **single point of failure** — if that one powerful server goes down, your entire app goes with it.

## 9. Horizontal Scaling (Scaling Out)

Horizontal scaling means adding more servers running the same code, then distributing traffic between them using a **load balancer**.

```
Before scaling:                    After scaling out:

                                   ┌─────────────┐
                                   │  Server 1   │
┌─────────────┐                  ┌─┤  (Node.js)  │
│  Server     │       ──────►   LB│  └─────────────┘
│  (Node.js)  │       ──────►   ──┤  ┌─────────────┐
└─────────────┘                  └─┤  Server 2   │
                                   │  (Node.js)  │
                                   └─────────────┘
                                   ┌─────────────┐
                                   │  Server 3   │
                                   │  (Node.js)  │
                                   └─────────────┘
```

A **load balancer** sits in front of all the servers and decides which one handles each incoming request. From the user's perspective, nothing is different — they still hit one URL and get a response. Behind the scenes, that request was handled by one of many servers.

### How it works in practice

Cloud platforms make horizontal scaling relatively straightforward. On AWS you can configure an **Auto Scaling Group** that automatically adds EC2 instances when CPU usage spikes above a threshold and removes them when traffic drops. You pay only for what you use.

### When it works well

- Your app handles a lot of concurrent users
- Traffic is unpredictable or spiky (e.g. a sale, a product launch, a viral moment)
- You need high availability — if one server goes down, the others keep serving users
- You want to scale specific parts of your system independently (more on this below)

### The key constraint: statelessness

For horizontal scaling to work properly, each server needs to be able to handle any request independently — it can't rely on data stored in memory from a previous request. This is called being **stateless**.

If your backend stores session data in memory (e.g. a variable that tracks who is logged in), that data lives on one server. The next request from the same user might go to a different server that has no idea about that session. This breaks the app.

The solution is to store shared state outside of the servers — in a database or a cache like Redis — so every server can access it regardless of which one handled the previous request.

## 10. Autoscaling

Manually adding and removing servers is slow and error-prone. **Autoscaling** automates this entirely.

You define rules — for example:

- If average CPU usage goes above 70% for 5 minutes → add a server
- If average CPU usage drops below 20% for 10 minutes → remove a server

The cloud platform monitors your servers continuously and adjusts the count automatically, without any human involvement. Traffic spikes at 2am? New servers spin up in seconds. Traffic drops off? Servers shut down and you stop being charged for them.

```
Traffic over time:

Requests │        ▲▲▲
         │      ▲▲   ▲▲
         │    ▲▲       ▲▲
         │  ▲▲           ▲▲
         │▲▲               ▲▲▲▲▲▲
         └─────────────────────────── time

Servers  │    ↑↑ ↑           ↓↓
running  │  1  2 3 3 3 3 3 2 1 1 1
         └─────────────────────────── time

         Autoscaler adds and removes servers to match demand
```

This is one of the biggest advantages of cloud infrastructure over owning physical servers. With physical servers, you buy for your peak load and those machines sit idle the rest of the time. With autoscaling, you only use (and pay for) what you actually need at any given moment.

## 11. Why Microservices and Horizontal Scaling Go Together

This is where microservices architecture starts to make real sense at scale.

In a **monolith**, every part of your app is deployed together as one unit. If your video encoding is under heavy load, you have to scale the entire app — including the parts that are completely fine. You end up running three copies of your user authentication code just because you needed more video processing capacity. That wastes resources.

In a **microservices** app, each service is deployed independently. You can scale just the parts that need it:

```
Normal traffic:                   High traffic (movie release weekend):

[Auth Service]      x1            [Auth Service]          x1
[User Service]      x1            [User Service]          x2
[Video Streaming]   x2            [Video Streaming]       x12  ◄── scaled up
[Recommendations]   x1            [Recommendations]       x4   ◄── scaled up
[Payments]          x1            [Payments]              x1
```

Netflix runs over 15,000 microservices across multiple cloud regions, and their architecture automatically scales individual services based on real-time demand. When a popular new show drops, the video streaming and recommendation services scale up aggressively — the payments service doesn't need to change at all.

This is efficient, targeted, and cost-effective in a way that a monolith simply cannot match at that scale.

---

## 12. What This Looks Like at Different Scales

| Scale | Architecture | Scaling approach |
|---|---|---|
| **Your project right now** | Monolith on one server | None needed yet — one Render instance is fine |
| **Small startup (thousands of users)** | Monolith | Vertical scaling + some horizontal scaling |
| **Growing product (hundreds of thousands of users)** | Monolith or early microservices | Horizontal scaling + autoscaling |
| **Large company (millions of users)** | Microservices | Per-service horizontal autoscaling across multiple cloud regions |

The pattern is consistent across the industry: startups often begin with a vertically scaled monolith, and as traffic grows they break services into microservices and scale out horizontally.

You don't design for Netflix scale on day one. You build a monolith, ship it, and scale when the problem actually exists.

## 13. Key Terms

| Term | Meaning |
|---|---|
| **Scaling** | Adjusting infrastructure to handle more or less load |
| **Vertical scaling** | Making one server more powerful (more CPU, RAM) |
| **Horizontal scaling** | Adding more servers and distributing traffic between them |
| **Load balancer** | A service that distributes incoming requests across multiple servers |
| **Autoscaling** | Automatically adding or removing servers based on traffic levels |
| **Stateless** | A server that doesn't rely on local memory between requests — required for horizontal scaling |
| **Single point of failure** | A part of the system that, if it goes down, takes everything with it |
| **Instance** | A single running copy of a server on a cloud platform |
