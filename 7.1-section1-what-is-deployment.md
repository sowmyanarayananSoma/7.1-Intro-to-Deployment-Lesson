# 7.1 – Section 1: What Is Deployment?

---

## 1. The Big Picture

You have been building apps on your own computer. The frontend, backend, and database all run on `localhost` — only you can access them.

**Deployment** is the process of moving your app off your machine and onto a server that is always on and connected to the internet, so that anyone in the world can use it.

The architecture of your app does not change. The same three pieces — frontend, backend, database — still exist and still talk to each other in exactly the same way. The only difference is *where* they are running.

---

## 2. Deployment Environments

Most apps don't go straight from your laptop to the public. They pass through a series of **environments** — each one a separate place where the app runs for a specific purpose.

The standard flow looks like this:

```
Local → Development → Staging → Production (Live)
```

| Environment | Who uses it | Purpose |
|---|---|---|
| **Local** | You | Write and test code on your own machine |
| **Development** | The dev team | A shared server for testing work in progress together |
| **Staging** | Dev team + QA | A near-identical copy of production for final testing before go-live |
| **Production** | Real users | The live app — real data, real traffic |

This is called the **"left to right" deployment model**. Changes start on the left (local) and are pushed rightward through each environment until they reach production. Nothing goes live without passing through the environments before it.

> You don't always need all four. A solo project might go straight from local to production. A large team always uses all four — and sometimes more.

---

## 3. The Deployment Process Flow

Deployment isn't just clicking an upload button. It follows a repeatable process every time.

### Step 1 — Plan

Before writing a single line of code, have a plan. Know what you're changing, how it affects the rest of the app, and when it will be deployed. For small bug fixes this takes a minute. For large features it might take days.

A deployment plan should include:
- What is being changed and why
- Which environments it will pass through
- Who is responsible for each step
- When it will go live

### Step 2 — Develop

Write your code in a **local or development environment** — never directly on production. Use **branches** in Git so your work-in-progress doesn't interfere with other developers' work.

### Step 3 — Test

Testing happens at every environment, not just at the end.

- On **local/dev**: Does the feature work as intended?
- On **staging**: Does it work in an environment that mirrors production? Are there any conflicts with other recent changes?

If a bug is found at any stage, the change goes back to development, gets fixed, and works its way through the environments again.

### Step 4 — Deploy to Production

Once everything passes testing on staging, the change is deployed to the live environment. At this point real users will see it.

Before pushing, do a final review of exactly what is different between your current live environment and what you're about to deploy. This step catches mistakes that somehow survived all the earlier testing.

### Step 5 — Monitor

Deployment doesn't end when you push to production. Watch the app after release. Real users will do things you didn't anticipate in testing. Monitor for errors, unexpected behaviour, and performance issues.

If something breaks, stay calm. Check whether a rollback is even possible and whether it would actually fix the issue before doing anything. Having a plan for this moment — before it happens — makes all the difference.

---

## 4. When Should You Deploy?

Timing matters. Two questions to ask:

1. **When do you have the fewest active users?**  
   Look at your analytics. Find the lowest-traffic window — often late at night or early morning. Fewer users online means fewer people affected if something goes wrong.

2. **Do you have developers available to fix things at that time?**  
   A 3am deployment with no one on call is risky. Find the best balance between low traffic and developer availability.

There is no perfect time to deploy, but there are clearly better and worse times.

## 5. Deployment Best Practices

### Use Git
Your code must be in version control. Without it you cannot track what changed, collaborate safely, or roll back if something breaks. Every deployment workflow depends on Git.

### Work in Branches
Never work directly on `main`. Use branches so multiple developers can work simultaneously without stepping on each other. A common pattern:

```
main       → maps to production (live)
dev        → maps to development environment
feature/*  → individual features in progress
```

When a bug needs fixing urgently, a developer can branch off `dev`, fix it, and merge — without disturbing any unfinished feature work.

### Work Locally First
Do your development and initial testing on your own machine before pushing anywhere. Local development is faster, doesn't require internet connectivity, and lets you iterate quickly without committing and deploying just to check if a change works.

### Use Environment Variables
Sensitive values — database connection strings, API keys, secret keys — must never be hardcoded in your code or committed to GitHub. Store them in a `.env` file locally and configure them separately in each environment's settings.

```
# .env  ← never commit this file
MONGO_URI=mongodb+srv://...
JWT_SECRET=supersecretkey
PORT=5000
```

### Keep `.env` in `.gitignore`
This is non-negotiable. If a `.env` file is ever pushed to a public GitHub repository, your credentials are exposed.

### Do a Final Review Before Going Live
Even after thorough testing, do one last review of exactly what is changing before hitting deploy on production. This catches the mistakes that somehow survived all earlier checks.

### Have a Deployment Schedule
Know when deployments happen and communicate it to the team. Scheduled deployments mean everyone knows what's coming, developers can be available to monitor, and surprises are reduced.

---

## 6. What Deployment Looks Like in Practice

Here is what a typical first deployment of a full-stack app involves, step by step:

1. **Code is pushed to GitHub** via `git push`
2. **The deployment platform pulls the code** from your repository
3. **Dependencies are installed** — `npm install` runs on the server
4. **The app is built** — for React, `npm run build` creates an optimized production bundle
5. **Environment variables are set** on the platform (your `.env` values, entered through a dashboard)
6. **The backend process starts** — equivalent to `node server.js`
7. **The database connects** — your app connects to a hosted database (e.g. MongoDB Atlas), not `localhost`
8. **A public URL is assigned** — your app is live

## 7. Key Terms

| Term | Meaning |
|---|---|
| **Deployment** | Moving an app to a server so others can access it |
| **Environment** | A place where the app runs — local, dev, staging, or production |
| **Production** | The live environment used by real users |
| **Staging** | A pre-production environment used for final testing |
| **Version control** | A system (Git) that tracks changes to your code over time |
| **Branch** | An isolated copy of your code for working on a feature or fix |
| **Environment variable** | A value stored outside your code, like an API key or DB connection string |
| **Rollback** | Reverting production to a previous version when something breaks |
| **Monitoring** | Watching a live app for errors and unexpected behaviour after deployment |
