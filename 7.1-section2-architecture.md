# 7.1 – Section 2: Architecture & Deployment

---

## 1. Your App Has an Architecture

You have already built a full-stack application. Even if you never called it this, it has an **architecture** — a structure that describes the parts of the system and how they communicate.

For the apps you have built in this course, the architecture looks like this:

```
┌─────────────┐        HTTP requests        ┌─────────────┐        queries        ┌─────────────┐
│             │  ─────────────────────────► │             │ ────────────────────► │             │
│   Frontend  │                             │   Backend   │                       │   Database  │
│   (React)   │ ◄───────────────────────── │  (Express)  │ ◄──────────────────── │  (MongoDB)  │
│             │        JSON responses       │             │        results        │             │
└─────────────┘                             └─────────────┘                       └─────────────┘
```

- The **frontend** is what users see and interact with in the browser
- The **backend** handles business logic and exposes API endpoints
- The **database** stores and retrieves data

Right now, all three of these run on your machine. They talk to each other over `localhost`. Nobody else can reach them.

---

## 2. Moving From Local to Public

To make your app accessible to the world, each of those three pieces needs to move off your machine and onto servers that are always on and publicly reachable.

The architecture stays exactly the same. The only thing that changes is *where* each piece runs:

```
                        ┌──────────────────────────────────────────────────────┐
                        │                   Cloud Platform                     │
                        │                                                      │
    User's browser      │  ┌───────────┐    ┌───────────┐    ┌─────────────┐  │
         │              │  │           │    │           │    │             │  │
         │   internet   │  │ Frontend  │───►│  Backend  │───►│  Database   │  │
         └─────────────►│  │  (React)  │    │ (Express) │    │  (MongoDB   │  │
                        │  │           │◄───│           │◄───│   Atlas)    │  │
                        │  └───────────┘    └───────────┘    └─────────────┘  │
                        │                                                      │
                        └──────────────────────────────────────────────────────┘
```

The user's browser sends a request over the internet to the deployed frontend. The frontend calls the deployed backend. The backend queries the hosted database. The response travels back the same way.

## 3. What Services Would You Deploy Each Part To?

Different parts of your app are deployed to different services, because they have different needs.

| Part | What it needs | Example services |
|---|---|---|
| **Frontend** (React) | Serve static files (HTML, CSS, JS) | Render (Static Site), Netlify, Vercel |
| **Backend** (Express) | Run a Node.js process 24/7 | Render (Web Service), Railway, Fly.io |
| **Database** (MongoDB) | Persistent cloud-hosted storage | MongoDB Atlas |

You can look up what technologies any real website is using at **[builtwith.com](https://builtwith.com)**. Type in any URL and it will show you the hosting, CDN, analytics tools, frameworks, and more that the site uses. It's a useful way to see how real companies structure their deployments.

> Note: BuiltWith shows you *what* a site uses, not *how* it's configured. It's great for exploration but won't show you the full architecture of complex systems.

## 4. Monolith vs. Microservices

There are two broad approaches to how you structure and deploy an application.

### Monolithic Architecture

A **monolith** is a single deployable unit. All parts of the application — frontend, backend, business logic — are packaged and deployed together.

```
┌──────────────────────────────┐
│         Single Deploy        │
│                              │
│  Frontend + Backend + Logic  │
│                              │
└──────────────────────────────┘
         │
         ▼
    One server
```

**Characteristics:**
- Simpler to build, deploy, and debug
- All code lives in one place
- Scaling means scaling the whole thing, even if only one part is under heavy load
- A bug in one part can bring down the whole app

Good for: early-stage projects, small teams, most apps you will build in your career.

---

### Microservices Architecture

A **microservices** app is split into many small, independent services. Each service handles one specific responsibility and is deployed separately.

```
┌──────────────┐   ┌──────────────┐   ┌──────────────┐   ┌──────────────┐
│   Auth       │   │   Products   │   │   Payments   │   │  Notifications│
│   Service    │   │   Service    │   │   Service    │   │   Service    │
└──────────────┘   └──────────────┘   └──────────────┘   └──────────────┘
       │                  │                  │                   │
       └──────────────────┴──────────────────┴───────────────────┘
                                    │
                              API Gateway
                                    │
                               Frontend
```

**Characteristics:**
- Each service can be built, deployed, and scaled independently
- A failure in one service doesn't necessarily take down the others
- Much more complex to build, test, and maintain
- Requires more infrastructure and DevOps knowledge

Good for: large teams, large-scale systems where different parts have very different traffic patterns.

---

### Netflix: A Real Microservices Example

Netflix is one of the most cited examples of microservices architecture. Looking at their tech stack, you can see how many independent systems are involved in running a single platform:

| Layer | What it does | Technologies used |
|---|---|---|
| **Frontend** | Web and mobile interfaces | React (JavaScript), iOS (Swift), Android (Kotlin) |
| **Backend** | Application servers and service logic | Spring Boot, Netflix Zuul, Netflix Eureka |
| **Databases** | Data storage across services | MySQL, Cassandra, CockroachDB, EVcache |
| **Streaming** | Video delivery at scale | Netflix CDN (CloudFront), Amazon S3 |
| **Messaging** | Async communication between services | Apache Kafka, Apache Flink |
| **Big Data** | Analytics and data processing | Amazon Redshift, Amazon S3, Apache Spark, Tableau |
| **CI/CD** | Deployment pipeline and monitoring | Jenkins, Jira, Confluence, PagerDuty, Netflix Atlas |

Each of these is a separate deployed service. Teams of engineers work on each one independently. When Netflix updates its recommendation algorithm, they don't redeploy the video streaming service — they only deploy the service that changed.

**The key point:** Netflix didn't start with this. They started as a monolith and evolved into microservices as their scale required it. You do not need microservices to ship a good product — you need them when your monolith can no longer handle the load or the team size.

## 5. What Does This Mean for Your App?

Take the To-Do List app you built in the backend unit. On your machine it looked like this:

- `localhost:5173` — React frontend
- `localhost:5000` — Express API
- `localhost:27017` — MongoDB

To make it public, each of those three addresses needs to be replaced with a real public URL:

| Before (local) | After (deployed) |
|---|---|
| `localhost:5173` | `https://my-todo-app.onrender.com` |
| `localhost:5000` | `https://my-todo-api.onrender.com` |
| `localhost:27017` | `mongodb+srv://...@cluster0.mongodb.net/todos` |

Your frontend code needs to point to the deployed backend URL. Your backend code needs to use the hosted database connection string. Both of those values should come from **environment variables** — not be hardcoded.

## 6. Key Terms

| Term | Meaning |
|---|---|
| **Architecture** | The structure of an app — its parts and how they connect |
| **Monolith** | An app deployed as a single unit |
| **Microservices** | An app split into many independently deployed services |
| **Static site** | A frontend with no server-side logic — just HTML, CSS, and JS files |
| **Web service** | A running server process (like your Express backend) |
| **CDN** | Content Delivery Network — serves static files from locations close to the user |
| **API Gateway** | A single entry point that routes requests to the correct microservice |
